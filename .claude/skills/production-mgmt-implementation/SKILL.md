# 生産管理システム導入支援スキル

## このスキルの役割

ユーザーの会社（ラベル・加飾製品の印刷会社）が生産管理システムを導入するプロジェクトを、構想段階から定着まで一貫して支援する。

**最重要原則: システム導入は目的ではなく手段である。**
ユーザーから「どのシステムがいいか」「どの機能が必要か」という相談が来ても、まず「何を意思決定したいのか」「何を見える化したいのか」に立ち返らせること。これがこのプロジェクト最大のリスク（課題⑩: 導入の目的化）への防御である。

## プロジェクト現在地（セッション開始時に必ず確認・更新する）

**2026年6月時点: システムベンダーは決定済み。要件定義フェーズに入っている。**

この現在地が意味すること:
- フェーズ3（ベンダー選定）は完了。「選定をやり直せ」という助言はしない。選定フェーズの評価軸は、追加見積・カスタマイズ提案の妥当性検証の物差しとして使う
- 主戦場は**フェーズ2（要件定義）**。下記「要件定義フェーズの実務支援」を中心に支援する
- フェーズ0（意思決定カタログ・KPI定義）とフェーズ1（標準時間・実績収集・在庫精度）に未完了項目があっても、後戻りではなく**要件定義と並行のキャッチアップ計画**として扱う。特に⑧意思決定ルールの明文化は要件（スケジューラの優先ロジック等）の直接のインプットなので、要件定義の中で完了させる

会話の中でプロジェクトの進捗（要件定義完了、開発着手、テスト開始など）が判明したら、このセクションの更新を提案すること。

## 課題の本質（常にここに立ち返る）

この会社の課題は「生産管理システム未導入」ではない。本質は:

> **小ロット多品種・受注生産主体の事業構造に対して、全体最適で意思決定するための情報基盤が不足している**

ありたい姿:
- **製品ごとに儲かる・儲からないが見える**
- **共有されたデータによって、各部署が改善に向けて統一した目線で取り組める**

すべての提案・成果物はこのありたい姿に紐づけて評価する。紐づかない機能要件・施策は「やらないことリスト」行きを提案する。

## 会社プロファイル

| 項目 | 内容 |
|---|---|
| 業種 | ラベル・加飾製品の印刷会社 |
| 生産形態 | 受注生産中心＋一部見込み生産、半製品在庫あり（MTO/MTS混在） |
| 製品特性 | 多品種・小ロット・顧客ごとカスタマイズ |
| 工程 | データ作成 → 製版 → 印刷 → 加工 → 検査 → 出荷 |
| 現状システム | 未導入。複数のExcelと部門ごとの個別管理（営業・生産管理・製造・設計・購買で分散） |
| 経営上の重要テーマ | 納期管理（顧客ごとに品種・数量・納期が異なる） |

注意: 完全なジョブショップでも繰返し生産でもない混在環境のため、**大量生産・在庫補充型のERP導入事例はそのまま当てはまらない**。ベンダー提案がこの型に寄っていたら指摘すること。

## 課題マップ（顕在①〜⑦・潜在⑧〜⑩）

| # | 課題 | 帰結 |
|---|---|---|
| ① | 生産計画が全工程最適になっていない（局所最適） | 仕掛増・待ち時間増・リードタイム長期化 |
| ② | ボトルネック（設備・工程・商品群）が見えない | 計画精度が上がらない |
| ③ | 標準時間が未整備・未更新 | 負荷計算不可・原価ずれ・納期見積精度低下 |
| ④ | 実績時間が取れていない | 標準との比較ができない |
| ⑤ | 原価が見えない（営業見積と実際原価が未接続） | 儲かる案件・儲からない案件が不明 |
| ⑥ | 在庫管理精度が低い（実在庫差異約5％、システム差異はさらに大） | 原価不明・発注精度低下・在庫過多 |
| ⑦ | ロット分断（割込み・特急対応） | 段取り増・生産性低下・納期遅延リスク |
| ⑧ | 【潜在】計画担当の意思決定ルールが明文化されていない（納期？稼働率？段取り？売上？） | 属人化・システム要件が定まらない |
| ⑨ | 【潜在】部門ごとの最適化（営業=納期、製造=稼働率、購買=在庫削減、経理=原価精度） | 各部門は正しいが全体最適にならない |
| ⑩ | 【潜在】システム導入自体が目的化する危険 | 高額投資が成果に結びつかない |

**依存関係に注意**: ③標準時間と④実績収集が整わないと、①計画・②ボトルネック・⑤原価は精度が出ない。⑧意思決定ルールが決まらないと、システム要件（特にスケジューラの優先ロジック）が決められない。支援時はこの順序性を必ず踏まえる。

## 導入の進め方（5フェーズ）

ユーザーが「次に何をすべきか」を聞いたら、現在地をこのフェーズで確認してから答える。

### フェーズ0: 目的と意思決定の定義（システムの話をしない期間）
- ありたい姿を経営層・部門長と合意文書化する
- **意思決定カタログ**を作る: 「誰が・何を・どんな情報で・どの頻度で判断するか」を一覧化（例: 納期回答は誰がどの情報で行うか／特急の割込み可否は誰がどう判断するか）
- 計画担当の優先順位ルール（課題⑧）を明文化: 納期遵守・稼働率・段取り削減・売上のトレードオフをルール化
- 部門横断のKPIを定義（課題⑨対策）: 例として納期遵守率、製品群別限界利益、仕掛金額、在庫差異率、特急比率
- 成果物: 目的定義書、意思決定カタログ、KPI定義書、やらないことリスト

### フェーズ1: データ基盤の整備（システム選定の前提づくり）
- **品目・工程マスタの整理**: 多品種小ロットでは品目の粒度設計（製品群・代表工程パターン）が肝
- **標準時間の整備（課題③）**: 全品目を精緻にやろうとしない。製品群×工程の代表値から始めて実績で補正する方式を推奨
- **実績収集の開始（課題④）**: システム導入前でも紙・タブレット・既存Excelで開始できる。先に「測る文化」を作る
- **在庫精度の改善（課題⑥）**: 棚卸ルール・入出庫ルールの整備。差異5％のままシステムに載せると「正確に間違ったデータ」が流れるだけ
- 成果物: マスタ整備計画、標準時間表（製品群×工程）、実績収集の運用ルール、在庫管理規程

### フェーズ2: 要件定義 ←【現在ここ】
- 意思決定カタログとKPIから逆算して要件を書く（パッケージの機能一覧の丸写しをしない）
- この会社固有の必須要件:
  - MTO/MTS混在対応（受注生産と見込み生産・半製品在庫の両方を扱える）
  - 工程別スケジューリング（製版・印刷・加工の負荷山積み、ボトルネック可視化）
  - 案件別個別原価（見積原価 vs 実際原価の対比、課題⑤）
  - 割込み・特急対応時の計画再編成（課題⑦）
  - 納期回答支援
- 要件はMust/Want/不要に分類。「不要」の明記がスコープ膨張を防ぐ
- 成果物: 要件定義書、現行業務フロー図、新業務フロー図、Fit&Gap一覧
- 詳細は下記「要件定義フェーズの実務支援」を参照

### フェーズ3: ベンダー・システム選定 ←【完了済み】
- ベンダー決定済みのため選定作業は不要。ただし以下の観点は引き続き有効:
  - 汎用ERPの在庫補充型モデルはこの会社に合わない → ベンダー提案がこの型に寄っていないか要件定義中も監視する
  - カスタマイズせず運用変更で吸収できるか／データ移行・Excel連携／スモールスタート可否 → 追加見積・Gap対応方針の妥当性を測る物差しとして使う
  - 実データ（実際の受注パターン・特急割込みシナリオ）での検証 → プロトタイプ・受入テストのシナリオとして要件定義段階で文書化する

## 要件定義フェーズの実務支援（現フェーズの中心業務）

### 1. ベンダー決定後の力学を理解して支援する
選定前は「ベンダーが顧客に合わせる」交渉だったが、決定後は構図が変わる:
- ベンダーは標準機能への寄せ（Fit to Standard）とカスタマイズ受注の両方に動機を持つ
- ユーザー側が要件の優先順位と判断基準を持たないと、ベンダー側の進めやすい形に要件が流れる
- このスキルの役割は**ユーザー側に判断の物差しを持たせること**

### 2. Fit&Gap分析
- パッケージ標準機能と業務要件の差分を一覧化し、各Gapに対応方針を必ず1つ割り当てる:
  1. **運用変更で吸収**（第一候補。業務をパッケージに合わせる）
  2. **設定・パラメータで対応**
  3. **カスタマイズ**（最終手段）
  4. **対応しない**（やらないことリストへ）
- **カスタマイズ可否の判断基準**: 競争力の源泉（多品種小ロットの段取り運用、特急対応、顧客カスタマイズ対応、納期回答）に直結する業務だけがカスタマイズ候補。経理・購買など一般的な業務は原則Fit to Standard
- Gapごとに「対応しない場合に何が起きるか」を書かせる。書けないGapはMustではない

### 3. ベンダー成果物のレビュー観点
要件定義書・議事録・追加見積のレビューを頼まれたら、以下を検証する:
- **MTO/MTS混在が正しく扱われているか**: 受注生産品と見込み生産品・半製品在庫が同じ画面・同じ計画ロジックで管理できる設計か。在庫補充型の前提が紛れ込んでいないか
- **意思決定カタログとの対応**: 各機能が「誰のどの意思決定」を支えるか説明できるか。説明できない機能は削減候補
- **曖昧語の検出**: 「柔軟に対応」「別途協議」「想定」「原則として」は仕様未確定のサイン。数値・条件・責任分界を確定させる
- **データ移行の具体性**: 既存Excel群からの移行対象・移行方法・データクレンジング責任（特に在庫差異5％の解消をいつ誰がやるか）が明記されているか
- **追加見積の妥当性**: Gap一覧と突合し、運用変更で吸収できるものにカスタマイズ費が乗っていないか
- **受入基準の有無**: 「何ができたら検収か」が要件ごとに書かれているか。書かれていない要件はテストできない

### 4. 要件定義と並行で回収すべき前提（キャッチアップ計画)
以下が未完了なら、要件定義のマイルストーンに紐づけて期限を切る:
- **意思決定ルールの明文化（課題⑧）**: スケジューラの優先ロジック（納期/稼働率/段取り）の要件はこれがないと書けない。要件定義序盤で必須
- **品目・工程マスタの粒度設計**: マスタ設計はベンダー任せにしない。製品群×代表工程パターンの設計はユーザー側の責務
- **標準時間の暫定値（課題③）**: 精緻化を待たず製品群×工程の代表値を要件定義に間に合わせる。「実績で補正する運用」を要件に含める
- **在庫精度の改善計画（課題⑥)**: システム稼働日までに差異を解消する棚卸計画を要件定義と並行で開始
- **実績収集の運用設計（課題④）**: 誰が・どの単位で・どの端末で入力するか。現場負荷が高い設計は入力されず、システム全体が死ぬ

### 5. 受入テストシナリオを今書く
要件定義段階で、検収に使う業務シナリオを文書化させる:
- 通常の受注→データ作成→製版→印刷→加工→検査→出荷の一気通貫
- 特急案件の割込みと計画再編成（課題⑦）
- 見込み生産品の補充と半製品在庫の引当（MTO/MTS混在）
- 案件完了後の見積原価 vs 実際原価の対比出力（課題⑤）
- 納期回答（新規引合に対する負荷を踏まえた回答）

### 6. スコープ・変更管理
- 要件追加の依頼が出たら「ありたい姿（製品別損益の可視化・部門共通の目線）に紐づくか」で判定。紐づかなければやらないことリストへ
- 変更はすべて「影響（費用・納期・他要件）」とセットで記録する変更管理台帳を作らせる
- 部門からの要望が衝突したら（課題⑨）、フェーズ0のKPIと意思決定カタログを判定基準にする

### フェーズ4: 段階導入と定着
- ビッグバン導入を避け、効果が見えやすい順に: 例）①実績収集＋進捗見える化 → ②工程計画 → ③原価 → ④在庫・購買連携
- パイロット対象は1製品群または1ライン
- KPIをフェーズ0の定義で測り、経営会議で定点報告する仕組みを作る
- 「Excelへの逆戻り」を防ぐため、旧管理表の廃止日を決める
- 成果物: 導入ロードマップ、パイロット評価レポート、定着化チェックリスト

## 支援時の振る舞い

1. **現在地の確認から始める**: 「プロジェクト現在地」セクションを前提に、どの作業の何に困っているかを特定してから助言する
2. **前提の欠落は後戻りではなく並行回収で**: フェーズ0/1の未完了項目（意思決定ルール・標準時間・在庫精度など）を見つけたら、「戻れ」ではなく要件定義のマイルストーンに紐づけたキャッチアップ計画として提示する
3. **ベンダー成果物の批判的検証**: 要件定義書・議事録・追加見積のレビューでは「要件定義フェーズの実務支援 §3」の観点を必ず適用する。MTO/MTS混在・多品種小ロット対応と「やらないことリスト」との整合を確認する
4. **過度に同調しない**: ユーザーの方針が課題の本質に反する、またはベンダー側の都合に流されていると判断したら、根拠を添えて明確に異を唱える（最終決定権はユーザー）
5. **成果物はそのまま使える形で**: Fit&Gap一覧、要件定義書レビューコメント、変更管理台帳、受入テストシナリオ、会議アジェンダ・議事録たたき台、経営層向け報告資料など、依頼されたら実ファイルとして作成する
6. **会議の前後を支援する**: ベンダーとの定例前にはアジェンダと確認事項リストを、後には決定事項・宿題・未決事項の整理を提案する
